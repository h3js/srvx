# TLS

> Serve over HTTPS and, optionally, require a client certificate (mutual TLS).

TLS (the successor of SSL) encrypts the connection between client and server using a **certificate** and its **private key**. Serving over HTTPS in srvx is a matter of providing them. **Mutual TLS (mTLS)** goes one step further and also asks the _client_ to present a certificate, which the server verifies against a trusted CA тАФ a common building block for service-to-service and zero-trust setups.

## Enabling HTTPS

Provide `tls.cert` and `tls.key`. srvx switches the [`protocol`](/guide/options#protocol) to `https` automatically.

```js
import { serve } from "srvx";

serve({
  tls: { cert: "./server.crt", key: "./server.key" },
  fetch: () => new Response("ЁЯСЛ Secure hello!"),
});
```

**[`tls`](/guide/options#tls) options:**

- `cert`: Certificate in PEM format тАФ file path or inline content (required).
- `key`: Private key in PEM format тАФ file path or inline content (required).
- `passphrase`: Passphrase for the private key (optional).

Server TLS works on **Node.js**, **Deno** and **Bun**.

> [!TIP]
> `cert`/`key`/`ca` values are treated as inline PEM when they start with `-----BEGIN `, otherwise as file paths.

> [!IMPORTANT]
>
> - Never commit private keys; load them from environment variables or a secret manager.
> - Consider automatic certificate management (e.g. Let's Encrypt) for production.

## Mutual TLS (mTLS)

Mutual TLS is provided by the opt-in `mtls()` plugin from `srvx/mtls`. It requests a client certificate during the handshake and exposes it тАФ with the negotiated protocol and cipher тАФ on [`request.tls`](#requesttls).

```js
import { serve } from "srvx/node";
import { mtls } from "srvx/mtls";

serve({
  tls: { cert: "./server.crt", key: "./server.key" },
  plugins: [mtls({ ca: "./ca.crt", requestCert: true })],
  fetch: (request) => {
    if (!request.tls?.authorized) {
      return new Response("client certificate required", { status: 401 });
    }
    return new Response(`Hello, ${request.tls.peerCertificate?.subject?.CN}`);
  },
});
```

**`mtls()` options:**

- `ca`: Trusted CA certificate(s) in PEM format тАФ file path(s) or inline content (optional). When set, replaces the well-known Mozilla CAs.
- `requestCert`: Request a certificate from connecting clients (default `true`).
- `rejectUnauthorized`: Reject clients whose certificate is not signed by a trusted `ca` (default `true`). When `false`, an unverified certificate is still exposed via [`request.tls`](#requesttls) with `authorized: false`.

### `request.tls`

While the plugin is active, [`request.tls`](/guide/handler#requesttls) provides:

- `peerCertificate` тАФ the client certificate ([node:tls `PeerCertificate`][PeerCertificate]). An empty object (`{}`) when none was presented.
- `authorized` тАФ `true` if the certificate was signed by a trusted `ca`.
- `authorizationError` тАФ why verification failed, when `authorized` is `false`.
- `protocol` тАФ negotiated TLS protocol, e.g. `"TLSv1.3"`.
- `cipher` тАФ negotiated cipher suite.

### Runtime support

> [!NOTE]
> `mtls()` requires srvx's **Node.js adapter** (`import { serve } from "srvx/node"`) and throws on any other server.
>
> - **Node.js** тАФ works natively.
> - **Deno** (2.8+) тАФ works through the Node adapter, which runs on Deno via `node:https`. (Native `Deno.serve` cannot request client certificates.)
> - **Bun** тАФ not currently supported: neither native `Bun.serve` nor Bun's `node:http(s)` server exposes the peer certificate to the handler, so the plugin throws. Tracked in [oven-sh/bun#16254](https://github.com/oven-sh/bun/issues/16254).

[PeerCertificate]: https://nodejs.org/api/tls.html#certificate-object
